# 문서(docs) 작성 원칙

이 폴더의 문서를 추가/수정하는 모든 에이전트는 아래 원칙을 따른다.

## 기능 문서 시스템 (Feature Docs)

기능 문서는 **개발자가 아닌 사용자를 독자로 상정한 기능 설명서**다. 코드 주석이나 설계 문서와 구분된다.

### 1. 위치와 폴더 구조

- 모든 기능 문서는 [`docs/features/`](./features/) 폴더 하위에 **기능별 '폴더'** 로 관리한다. (이하 **개별 기능 문서 폴더**)
- 개별 기능 문서 폴더 이름은 다음 규칙을 따른다:
  - **3자리 숫자 인덱스**를 접두사로 붙인다. (`001`, `002`, …)
  - **영문**으로 작성한다.
  - 인덱스와 기능명은 **하이픈**(`-`)으로 잇는다.
  - 기능명 내부의 공백은 **언더스코어**(`_`)로 잇는다.
  - 예: `001-editor_context_tag`
- 개별 기능 문서 폴더에는 해당 기능 문서의 **언어별 번역본들**을 저장한다. (예: `en.md`, `ko.md`, …)

### 2. 메타 파일 (색인)

- 개별 기능 문서 폴더들을 모아둔 [`docs/features/`](./features/) 아래에는 [`docs/features/CLAUDE.md`](./features/CLAUDE.md)(이하 **메타 파일**)가 있고, 개별 기능 문서 폴더들을 **색인**하는 역할을 포괄한다.
- 메타 파일에 기능을 색인할 때는 **기능별로 불릿포인트 마크다운 링크 한 줄**이면 충분하다.

### 3. 작성 톤과 내용

- 사용자를 독자로 상정하므로, **업데이트 노트와 유사한 톤앤매너**로 작성한다.
- 사용자에게 안내되어야 하는 내용을 **최대한 상세하면서도 쉽게** 작성한다.
- 관련된 **버전·PR·이슈** 등을 적극적으로 링크하여 맥락을 충분히 제공한다.

### 4. 작성 책임

- 모든 기능 PR은 가능한 경우라면 **반드시 기능 문서를 작성하거나 최신화**하기 위해 노력해야 한다.
- 사용자가 기능 문서 작성을 지시하지 않더라도, **기능 문서 작성을 적극적으로 제안**하여, 사용자가 명시적으로 거절하지 않는 한 작업을 놓치지 않도록 돕는다.
